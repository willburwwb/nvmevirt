### 实验设置

- 平台：AMD Ryzen 9 5950X，16 物理核 / 32 逻辑核，单 NUMA 节点。
- NVMeVirt
    - 1 个 dispatcher 线程，若干个 io_worker 线程。dispatcher 负责轮询 doorbell、扫描 SQ、根据设备模型计算目标完成时间，并将请求分发给 worker；worker 负责实际数据搬运和 CQ 写回。

- 观测指标：
    - fio 侧：IOPS、latency
    - NVMeVirt 侧：io_worker_latency, io_dispatcher_latency

### 实验结论

1. SPDK 与 nvmevirt 对比
    - 在 SSD 1-3 三组中，SPDK 与 io_uring 的结果较为接近。这说明在中低速设备模型下，host 侧软件栈不是主要瓶颈
    - 在高性能设备模型下，SPDK 在低并发时有更明显优势。例如 P5800X 配置下，单线程测试从 0.713M 提升到 0.935M IOPS。这说明 SPDK 降低了 host 软件栈开销
    - 但当线程数增加到 4~16 后，SPDK 与 io_uring 很快收敛到相近上限。例如 Optane 5M 配置下，两者都稳定在 1.7~1.8M IOPS
2. NVMeVirt 的实际瓶颈
    - 调整 io_worker 的数量，测试 io_uring 在 4K randread 下的  peak iops ( fio 使用 16 个线程)
        
        
        | number of worker  | 1 | 2 | 4 | 8 |
        | --- | --- | --- | --- | --- |
        | IOPS | 1.48M | 1.739M | 1.689M | 1.801M |
    - 可以看到 io_worker 从 1 增加到 2 能带来提升，但从 2 到 8 已几乎无收益
    - 进一步看 nvmevirt io_worker 的内部时延统计，4-worker 时单个 worker 每个 io copy 以及将 cq 写回去的时间为 705ns，理论每个 worker 的 iops 为 1.41M。而同一组实验中 fio 的端到端 clat_avg 却达到 700+us。这说明 worker 侧的数据搬运和排队并不是端到端时延的主要来源
    - 而在 dispatcher 侧的累计统计显示，在 4-worker 配置下，处理  127 个 io 请求需要 65.86 us。因此 dispatcher 阶段的理论处理能力约为：127 / 65.86us = 1.92M IOPS
    - 因此，NVMeVirt 的主瓶颈并不在 io_worker 上，而在单线程 dispatcher 对活跃 SQ 的串行处理路径上

孙老师您好，这部分实验我做了点测试，得到了一些初步结果。

从目前结果来看，当使用 NVMeVirt 模拟中低性能 SSD 时，例如 250K、500K IOPS 这类配置，SPDK 和 io_uring 的测试结果比较接近，二者都基本能够逼近设备模型的理论上限。这说明此时host 侧 I/O 软件栈的开销还不是主要瓶颈。

当进一步模拟较高性能 SSD 时，例如 1M、1.5M IOPS 配置，在相同 I/O 线程数下，SPDK 的 IOPS 会高于 io_uring。这说明随着设备性能提升，host 侧软件栈开销开始变得显著，SPDK 在降低软件栈开销方面是有效的。

不过，当继续把设备模型提升到更高配置，例如 2.5M、5M IOPS 时，无论是 SPDK 还是 io_uring，最终都只能跑到大约 1.7M~1.8M IOPS。结合排查来看，这里的主要瓶颈已经不在 host 软件栈，而是在 NVMeVirt 本身。

具体来说，虽然 NVMeVirt 可以配置多个 io_worker 线程处理 I/O 请求，但当前设计中只有一个 io_dispatcher 线程负责轮询 doorbell、扫描 SQ，并将请求分发给各个 worker。然后，实验表明 worker 侧的数据搬运和 CQ 写回开销并不是主要问题，真正限制整体吞吐的是 dispatcher 对活跃 SQ 的单线程串行处理路径。因此，如果希望进一步提升模拟器可达到的 IOPS 上限，可能需要从 NVMeVirt 的架构设计上做修改，比如支持多线程并行处理 I/O 分发。
